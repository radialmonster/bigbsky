You are managing Bigbsky changelog fragments.

Goal:
- Keep changelog entries small, issue-linked, and parseable.
- Avoid repeated edits to `changelog/internal/CHANGELOG.md`.
- Compile fragments into release notes or monthly archives only when asked.
- Roast changelog fragments before they become release notes.

Fragment format:
- Issue or PR id and short title.
- Done bullets.
- Public Notes for the app-facing changelog. Write 1-3 prose sentences per change describing what the user sees and why it matters. These drive the public changelog narrative, so avoid terse single-line bullets like "Fixed X" — instead describe the improvement in a way that reads naturally in a paragraph. Use `None` for internal tooling, workflow changes, investigations, admin-only operational changes, failures, upcoming features, or anything not visible/valuable to normal Bigbsky users.
- Verified bullets.
- Files touched bullets.
- Follow-up bullets when work is partial or adjacent issues were created.
- Hidden metadata comment:
  `<!-- bigbsky:changelog {"issue":23,"status":"needs-roast"} -->`

Rules:
- Do not include secrets.
- Do not include huge raw logs.
- Preserve UTF-8 text.
- Prefer concise product/operational summaries over commit-level noise.
- Treat `## Public Notes` as stricter than `## Done`: it must be safe for customers to read in the app. It must not mention implementation tooling, verification failures, release blockers, PR mechanics, internal planning, future promises, billing/admin work that ordinary tenant users should not see, or issue/process details.
- A fragment is ready only when it accurately summarizes the issue work, verification is real, files touched are specific, and follow-ups are explicit.
- If the fragment is weak, edit the fragment directly instead of adding release-note noise elsewhere.
- If the roast finds a real bug, missing verification, incomplete acceptance criterion, UX gap, security/privacy risk, or follow-up improvement, check existing open issues before creating anything. Reuse exact duplicates, amend/comment on same-scope issues with the new finding, and create a new related follow-up only when the finding is adjacent but separately actionable.
- New follow-up issues must be labeled `ai:needs-roast` and `ai:follow-up`, attached as a sub-issue of the original issue, and linked in the fragment Follow-up section.
- Every follow-up issue created from a fragment must include `Follow-up from Issue #N`, `Original issue: #N`, and the changelog fragment path so GitHub text references link the follow-up back to the original work. The automation should also create the GitHub Relationships/sub-issue link.
- Follow-up issue type should be `Bug` for failed verification, broken behavior, or incomplete prior work; `Feature` for newly discovered capability; and `Task` for docs/specs/workflow/refactor/tests/investigation.
- Reopen or keep the original issue only when its stated acceptance criteria were not actually met. Create a separate follow-up issue when the shipped work is valid but exposes adjacent work.
- Do not edit `changelog/internal/CHANGELOG.md` in fragment-roast mode.
