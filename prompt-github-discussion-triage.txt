Log the start time immediately and record the current date with `Get-Date -Format o`.

You are running the Bigbsky GitHub discussion-triage loop from the current repo/worktree. This is the discussion-side roast/refinement pass. Do not switch to `N:\Projects\bigbsky-github` when this prompt is running from a lane worktree.

Goal:
- Triage one GitHub Discussion labeled `ai:needs-triage`, or the oldest open discussion with no workflow state label. Discussions labeled `priority:urgent` are selected before non-urgent discussions.
- If the discussion is a brief human seed, investigate and extrapolate the likely product intent before deciding its state.
- If the discussion has no short-term/long-term planning label, decide whether it is likely `short-term` or `long-term` and add that label.
- If the idea is clear and sufficiently fleshed out, label it `ai:ready-to-promote` and leave a neutral workflow comment.
- If it needs a real product/business answer, ask the smallest useful question and label it `ai:needs-user-answer`.
- If it is too broad, needs splitting, or is still underdeveloped, keep `ai:needs-triage` so the loop can return and do the split work. Do not add `ai:blocked` for scope or splitting reasons.
- If it is blocked by a genuine external dependency or a decision that cannot be made without human input, label it `ai:blocked`.
- Stop after one discussion.

Rules:
- Do not edit application code.
- Do not edit `changelog/internal/CHANGELOG.md`.
- Do not put secrets into GitHub comments, issue bodies, discussion bodies, logs, or generated files.
- Do not ask the user questions unless the answer cannot be inferred from the discussion, existing Bigbsky specs, or common repair-shop workflow.
- Comments may appear under the repository owner's account. Use neutral wording such as `Workflow update:`.
- Discussion labels are separate from issue roast labels. Do not add `ai:needs-roast` or `ai:fully-roasted` to discussions; those belong to GitHub Issues after promotion.
- Respect an existing `short-term` or `long-term` label when the user already chose one. New GitHub-native discussions should use `short-term` or `long-term`.
- If the selected discussion is labeled `duplicate`, or its body or comments clearly identify it as a duplicate of an existing issue or discussion, add a `Workflow update:` comment referencing the original item (if identifiable), then close the discussion with reason `DUPLICATE` using the `closeDiscussion` GraphQL mutation. Do not spend triage effort on it.

Definition:
- Triage means converting a rough human planning note into a clear product/workflow understanding and next state.
- Treat short human notes as seeds, not finished requirements. Expand them into likely UI, UX, app workflow, data relationship, permissions, edge cases, and implementation-slice implications.
- Confirm the business/workflow intent, affected user, desired outcome, rough scope, obvious duplicates, and whether the idea should stay short-term or long-term.
- Short-term means the item appears actionable soon, supports current product workflows, fixes user friction, or can be promoted after normal issue roast. Long-term means it is broad, strategic, dependent on major foundation work, or should be deferred until later.
- Prefer making reasonable Bigbsky/product assumptions over asking the user.
- Ask the user only when a blocking business/workflow decision cannot be inferred.
- Do not mark a very brief seed `ai:ready-to-promote` until the discussion comment or body contains enough product understanding for issue promotion.

Process:
1. Use GitHub CLI GraphQL to find a discussion labeled `ai:needs-triage` that is not `ai:claimed`, `ai:promoted`, `ai:blocked`, or `ai:needs-user-answer`. Prefer `priority:urgent` discussions first, then oldest non-urgent discussions. If none exists, find the oldest open discussion with no workflow state label and treat it as needing triage, again preferring `priority:urgent` if present.
2. Add `ai:claimed` to that discussion.
3. Read the discussion body and comments.
4. Read `prompts/github/discussion-triage.txt` for the triage rubric.
5. Decide whether it is ready to promote, needs a user answer, should be split, needs more AI triage, or is blocked.
6. Update labels:
   - Always add either `short-term` or `long-term` when neither planning label is present.
   - Ready: add `ai:ready-to-promote`, remove `ai:needs-triage`, remove `ai:claimed`.
   - Needs user answer: add `ai:needs-user-answer`, remove `ai:needs-triage`, remove `ai:claimed`.
   - Needs more AI triage, too broad, or needs splitting: keep `ai:needs-triage`, remove `ai:claimed`. Do NOT add `ai:blocked` for scope or splitting reasons -- the triage loop must be able to return and complete the split.
   - Blocked by a genuine external dependency or a human-only decision: add `ai:blocked`, remove `ai:claimed`.
   - Preserve `priority:urgent` if present so promotion can carry urgency onto the created issue.
7. Add one neutral workflow comment explaining the decision.
8. Verify the final discussion labels/comments.

Helpful GraphQL operations:
- Query repository discussions, labels, and discussion IDs.
- Use `addLabelsToLabelable` and `removeLabelsFromLabelable` for discussion labels.
- Use `addDiscussionComment` for comments.

End:
- Log end time and elapsed minutes.
- Print the discussion number, final labels, and whether it is ready to promote.
