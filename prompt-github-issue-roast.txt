Log the start time immediately and record the current date with `Get-Date -Format o`.

You are running the Bigbsky GitHub issue-roast loop from the current repo/worktree. Do not switch to `N:\Projects\bigbsky-github` when this prompt is running from a lane worktree.

Goal:
- Take one GitHub issue labeled `ai:needs-roast`; if none are available, re-roast the oldest eligible open issue.
- Use the local automation to gather issue comments and relevant code/spec context.
- Roast the issue itself, not the user.
- Apply the improved issue body/labels/comment back to GitHub.
- Stop after one issue so concurrent loops stay small and conflicts are limited.

Rules:
- Do not edit application code.
- Do not edit `changelog/internal/CHANGELOG.md`.
- Do not put secrets in issue bodies, comments, prompt files, logs, or generated artifacts.
- Prefer defensible product/workflow decisions from Bigbsky context. Ask the user only when no safe default exists.
- GitHub comments may appear under the repository owner's account. Use neutral wording such as `Workflow update:`. Do not write `I updated`, `I decided`, or similar first-person wording.
- If the issue is not ready after this pass, keep `ai:needs-roast` and explain the remaining roast gap in the issue body/comment.
- Keep or add `ai:needs-roast` when this pass found and fixed any meaningful issue-quality gap.
- Add `ai:fully-roasted` and remove `ai:needs-roast` only on a pass where active critique finds no meaningful issue-quality gap. This is the label that makes the issue eligible for implementation.
- Populate `roast_findings` in the result JSON with the meaningful problems found during this pass. Use an empty array only when the issue was already clean after active critique.
- Set or correct the issue type when the current issue body makes it clear: `Feature` for new product capability, `Bug` for broken behavior or fixing previous work, and `Task` for specs/docs/workflow/refactor/tests/investigation.
- If the issue exists specifically to unblock or repair an existing open implementation PR (for example: roast detects this issue is a scope-split or follow-up that gates a named open PR, or the body references an existing PR that needs this work to merge), add `priority:urgent` so it jumps the roast queue. The queue sorts by `createdAt` oldest-first; without urgent the new unblock-issue queues behind every older needs-roast issue and the parent PR sits idle. Do NOT add urgent for non-blocking adjacent improvements.
- The apply script tracks roast count in hidden issue metadata. Do not manually edit that metadata.

Process:
1. Run:
   ```powershell
   .\scripts\github-loop\issue-roast.ps1 -Mode Prepare
   ```
2. Read the generated `out/issue-roast/issue-N/prompt.md`.
3. Produce JSON only using the shape requested in that prompt.
4. Write that JSON to `out/issue-roast/issue-N/result.json`.
5. Validate the JSON with `ConvertFrom-Json`.
6. Apply it:
   ```powershell
   .\scripts\github-loop\issue-roast.ps1 -Mode Apply -IssueNumber N
   ```
7. If apply reports that a new non-workflow comment arrived after prepare, rerun prepare for that same issue, reread the fresh prompt, regenerate `result.json`, and apply the fresh result. Do not force the stale result through.
8. Verify the issue labels/body/comment with `gh issue view N --repo radialmonster/bigbsky-dev`.
9. Print a concise summary: issue number, labels changed, whether it is fully roasted, and any remaining blocking user question.

Selection (the prepare step picks an issue in this exact order; every tier excludes `ai:claimed`, `ai:fully-roasted`, `ai:blocked`, `ai:infra-blocked`, and `ai:needs-user-answer`):
1. `priority:urgent` issues, preferring those labeled `ai:needs-roast` before urgent re-roasts.
2. `ai:blocks-release` issues, preferring those labeled `ai:needs-roast` before release-blocker re-roasts.
3. Ordinary first-pass `ai:needs-roast` issues, oldest first.
4. Re-roast of the oldest eligible open issue if no `ai:needs-roast` work remains.
- Issues already labeled `ai:fully-roasted` are skipped by issue-roast because they belong to implementation/work selection unless a stale human update returns them to `ai:needs-roast`.

Concurrency and side effects:
- The prepare step claims the issue with `ai:claimed`; respect that label.
- Do not work on an already-claimed issue.
- If the result JSON includes `blocked_by_issue_numbers`, the apply script will force-add `ai:blocked` and `ai:needs-roast`, strip `ai:fully-roasted`, and create native `blocked by` issue dependencies. Do not submit a result with `blocked_by_issue_numbers` non-empty AND `ai:fully-roasted` set true — the script will reject or override that combination.
- `ai:fully-roasted` must be paired with removing `ai:needs-roast` in the same apply. The script enforces this; do not request `ai:fully-roasted` while keeping `ai:needs-roast`.
- If no issue is available, print that no issue was available and stop.

End:
- Log end time and elapsed minutes.
